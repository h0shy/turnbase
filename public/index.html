How it works

The neutral referee problem

When two parties interact in any structured way, someone has to run the rules. Whoever runs them controls them. Turnbase is the substrate neither party controls.

⬡
Sessions
Every interaction is a session with a pinned engine version and config hash. The execution environment is fully specified before any player joins — preventing post-hoc rule changes.
◑
Observation scoping
Each participant receives only what they're allowed to see. In chess, that's the full board. In Kuhn Poker, it's your card only — the server enforces information boundaries between players.
◈
Verifiable receipts
Every state transition produces a signed receipt: action, state hashes before and after, engine version, config hash, timestamp. Any third party can independently replay the game from the transcript.
Quick start

Works with curl

No SDK required. The protocol is plain HTTP and JSON.

shell turnbase.app
# 1. Create a session SESSION=$(curl -s -X POST https://turnbase.app/sessions -H 'Content-Type: application/json' -d '{"engine":"chess"}' | jq -r '.sessionId') # 2. Two players join curl -X POST "https://turnbase.app/sessions/$SESSION/join" -d '{"playerId":"alice"}' curl -X POST "https://turnbase.app/sessions/$SESSION/join" -d '{"playerId":"bob"}' # 3. Submit a move curl -X POST "https://turnbase.app/sessions/$SESSION/actions" -d '{"playerId":"alice","action":{"from":"e2","to":"e4"}}' # 4. Get the signed receipt curl "https://turnbase.app/sessions/$SESSION/receipt/1"
Engines

Two engines, one protocol

The session, receipt, and transcript machinery is identical across engines. Only the rules change.

chess
Standard chess. Fully deterministic, no hidden information. First to join plays white. Validates all FIDE rules including en passant, castling, and promotion.
{"engine": "chess"}
{"from": "e2", "to": "e4"}
kuhn-poker
3-card simplified poker (J < Q < K). Proves observation ≠ state — each player sees only their own card. The server enforces information asymmetry. Seeded for reproducibility.
{"engine": "kuhn-poker", "config": {"seed": 42}}
{"type": "BET"}
Roadmap

What's next

Turnbase is moving toward on-chain verifiability and agent identity.

✓
Phase 1 — Chess engine
Deterministic rule enforcement with HMAC-SHA256 signed receipts. Full FIDE rules, hash chain integrity, and independent transcript replay.
✓
Phase 1.5 — Hidden information
Kuhn Poker engine proves observation scoping works — each player sees only their own card. The server enforces information asymmetry at the protocol level.
◌
Phase 2 — ERC-8004 identity
Register Turnbase as an on-chain agent via ERC-8004. Replace HMAC with EIP-712 signatures verifiable against the on-chain identity registry. Receipts become independently verifiable by anyone.
